# Feature PRD Output Template — Phase A

This file defines the exact structure for `/generate-prd --feature` (Phase A — Feature PRD) output.
Phase A focuses on feature definitions, user flows, and page-level specifications.
Technical sections (Schema, Permissions, System Design) are generated by Phase B (`/generate-prd --tech`, or chained automatically in default `--full` mode) and merged into the same `{App}_PRD.md` file.

> **⚠️ Template Maintenance Rules** — All editors (human or AI) modifying this template MUST follow these rules:
>
> 1. **Universality principle**: All structures, sections, fields, and checklists must remain generic and applicable to ANY project.
> 2. **No project-specific terms**: Do not embed project-specific terms (app names, domain feature names, specific service names) into the main structure.
> 3. **Examples only**: Project-specific content is ONLY allowed inside clearly labeled `Example:` blocks.
> 4. **Generalize first**: When adding a useful pattern discovered in a specific project, generalize it before adding.
>    - ❌ "Check exercise reminder notification" → project-specific
>    - ✅ "If scheduling feature exists, verify reminder notification trigger" → universal
> 5. **Conditional validation**: Write validation items as conditional checks based on feature existence (e.g., "If chat feature exists → verify message notification is defined").

> **TBD Handling**: Every unclear item gets `[💡 Recommended: X]` with rationale. PM confirms all items before Phase A is finalized.
> Accepted → `[📌 Adopted]` | Modified → `[📌 Adopted: {PM's version}]` | Rejected → Moved to Open Questions

> **8 Items Per Route**: Every route must have all 8 mandatory items. Use "N/A" with brief reason when an item does not apply.
> Example: `**2. Input Validation Rules**: N/A (no inputs on this page)`
