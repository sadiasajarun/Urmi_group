# Deepen User Stories

Iteratively enriches `.claude-project/user_stories/*.yaml` files by analyzing UI components for coverage gaps across 5 depth tiers (boundary conditions, auth role boundary, network errors, state-dependent scenarios, keyboard interaction). Stories accumulate across iterations via marker-based append — existing stories are never modified or deleted.

---

## When to Use

- After Phase 2c (base stories generated by `generate-user-stories`)
- As `--phase user-stories --loop` standalone pipeline command
- Automatically before Phase 9c acceptance testing in `--loop` mode
- When `STORY_COVERAGE.md` shows `gap_score >= 0.05` (coverage not yet saturated)

---

## Prerequisites

1. Base user story files exist: `.claude-project/user_stories/*.yaml` (generated by `generate-user-stories`)
2. Page UI source exists (one of):
   - **Phase 3+ (preferred when TSX not available)**: HTML prototypes at `.claude-project/design/variations/**/*.html`
   - **Phase 6+ (preferred when available)**: React components at `frontend/app/pages/**/*.tsx`
3. Routes manifest exists: `.claude-project/routes.yaml`

## Tier Taxonomy

### Base Tiers (1-8) — generated by `generate-user-stories`

| Tier | Name | Tag Pattern |
|------|------|-------------|
| 1 | Smoke | `[smoke]` |
| 2 | Form Submission | `[interaction, form]` |
| 3 | Form Validation | `[validation, form]` |
| 4 | Search & Filter | `[interaction, filter]` |
| 5 | CRUD Operations | `[interaction, crud]` |
| 6 | Pagination | `[interaction, pagination]` |
| 7 | Empty & Error States | `[validation, state]` |
| 8 | Cross-Page Flows | `[navigation, flow]` |

### Depth Tiers (9-13) — generated by this skill

| Tier | Name | Tags | Applicable When |
|------|------|------|-----------------|
| 9 | Boundary Conditions | `[negative, boundary]` | Page has input fields with `maxLength`, numeric `min`/`max`, or paginated lists |
| 10 | Auth Role Boundary | `[negative, auth]` | Route is admin-only or auth-required (test wrong role access) |
| 11 | Network/API Errors | `[negative, network]` | Page has `useQuery`/`useMutation` hooks with error handling (`isError`, `onError`) |
| 12 | State-Dependent | `[state, dependent]` | Page renders differently based on data presence (empty vs populated, logged in vs out) |
| 13 | Keyboard Interaction | `[accessibility, keyboard]` | Page has forms with multiple input fields or submit buttons |
| 14 | Known Bug Patterns | `[proactive, bug-pattern]` | Route matches high-frequency cross-project bug category |

### Tier Applicability Rules

A tier is **applicable** to a route only if the route's UI inventory contains matching elements:

```
Tier 9 applicable IF: page has input[maxLength], input[type=number], or pagination controls
Tier 10 applicable IF: route is in auth_required or admin_required section of routes.yaml
Tier 11 applicable IF: page .tsx has useQuery/useMutation with isError or onError handling
Tier 12 applicable IF: page .tsx has conditional renders based on data length/presence (e.g., {items.length === 0 && ...})
Tier 13 applicable IF: page has 2+ input fields in a form
Tier 14 applicable IF: route has matching bug_pattern_signals from Phase 0.7
  AND frequency >= 5 (always) OR frequency >= 3 (if route has thin coverage)
  AND NOT already covered by _gap_patterns.yaml for this route+category
```

## Workflow Language Extensions

### Extended Actions (new — added to canonical verb set)

- `Press Enter in {field-description}` — submits form via keyboard Enter key
- `Press Tab to focus {field-description}` — keyboard navigation between fields
- `Fill {field-description} with {n} characters of text` — generates a string of exactly n characters for boundary testing

### Extended Assertions (new)

- `Verify {button-description} is disabled` — checks the disabled state of a button/control
- `Verify {input-description} has maximum length {n}` — checks the `maxLength` attribute is enforced

All other workflow verbs follow the canonical set defined in `generate-user-stories/SKILL.md`.

## Algorithm

### Phase 0.3: Cross-Reference PRD Failure Cases

```
Read .claude-project/context/prd-failure-cases.md (pre-loaded in context).

IF file does not exist or is empty: skip silently → continue to Phase 0.5.

For each section in prd-failure-cases.md:
  Permission / Auth Failures     → map to Tier 10 (Auth Role Boundary)
  Validation / Input Failures    → map to Tier 9 (Boundary) + Tier 11 (Network/API Errors)
  Duplicate / Conflict Failures  → map to Tier 11 (Network/API Errors, 409)
  Limit / Boundary Conditions    → map to Tier 9 (Boundary Conditions)
  Empty / Not Found States       → map to Tier 12 (State-Dependent) + Tier 11 (404)
  Error Response Definitions     → map to Tier 11 (Network/API Errors)

For each extracted failure case:
  1. Match to route(s) in routes.yaml by feature/page name
  2. Check if existing stories already cover this failure
     IF covered: skip
     IF NOT covered: add to gap_list with source="prd-failure-cases"

Track: "PRD failure cases loaded: {N} permission, {M} validation, {K} boundary, {L} empty-state"
```

### Phase 0.5: Load Gap Patterns (Feedback Loop)

```
Read .claude-project/bug_reports/_gap_patterns.yaml if it exists.

IF file does not exist: skip silently → continue to Phase 1.

For each pattern in patterns[]:
  IF pattern.applicable_tiers includes any tier in range [9-14]:
    → Check if the pattern's detection_rule extends an existing
      depth tier's applicability condition.
    → If yes: add the new condition to that tier's applicability
      rules for Phase 3 gap analysis.
    → Example: if GP-005 says "Tier 12 should also apply when
      component has useReducer with multiple dispatch types",
      add that condition to Tier 12's check.
    → Track: "Extended Tier {N} applicability with GP-{id}"

  IF pattern.applicable_tiers includes any tier > 13:
    → These are new tiers proposed by the feedback loop.
    → Register as custom depth tiers for Phase 4 generation.
    → Apply the template from detection_rule when the condition is met.
    → Generated stories tagged: [feedback-loop, tier-{N}]
    → Track: "Registered custom Tier {N} from GP-{id}"

Log summary: "Loaded {N} gap patterns ({M} extend depth tiers, {K} propose new tiers)"
```

### Phase 0.7: Load Bug Pattern Knowledge Base

```
1. Read global patterns:
   skills/qa/learn-bugs/references/bug-patterns-global.yaml
   IF not found: WARN "No global bug patterns. Tier 14 will be empty.", continue.

2. Read project patterns (optional):
   .claude-project/knowledge/bug-patterns.yaml
   IF not found: skip (global-only mode)

3. Merge:
   - If both exist: project overrides global (by pattern ID)
   - If pattern has skip: true in project file: exclude

4. Filter:
   - frequency >= 5 (high): always include
   - frequency >= 3 (medium): include only if matching route has thin coverage
   - frequency < 3 (low): exclude

5. Build category_detection_map:
   For each category in filtered patterns, collect code_signals:
   {
     "auth": ["login", "signup", "auth", "token", "session"],
     "form_input": ["<form", "onSubmit", "<input", "validation", "useForm"],
     "list_pagination": ["pagination", "page", "limit", "offset"],
     "search_filter": ["search", "filter", "query", "autocomplete"],
     "file_upload": ["<input type=\"file\"", "FormData", "upload"],
     "payment": ["payment", "checkout", "stripe", "cart", "invoice"],
     "realtime": ["WebSocket", "socket.io", "EventSource"],
     "navigation": ["navigate", "router", "history", "useParams"],
     "data_sync": ["useQuery", "invalidate", "cache", "optimistic"],
     "push_notification": ["notification", "FCM", "push"],
     "admin_dashboard": ["admin", "dashboard", "statistics"],
     "performance": ["useMemo", "useCallback", "lazy", "Suspense"]
   }

6. Log summary: "Loaded {N} bug patterns across {M} categories for Tier 14 detection"
```

### Phase 1: Inventory Existing Coverage

```
1. Read all .claude-project/user_stories/*.yaml files (skip _fixtures.yaml)
2. For each file:
   a. Parse all stories
   b. For each story, read the `tags` field
   c. Map tags to tier numbers using the Tier Taxonomy table above
   d. Record: { file, url, tiers_covered: Set<tier_number> }
3. Build coverage map:
   coverage_map = {
     "home.yaml": { url: "/", tiers: [1, 2, 4, 8], story_count: 10 },
     "products.yaml": { url: "/products", tiers: [1, 2, 3, 4, 5, 6, 7], story_count: 18 },
     ...
   }
4. If STORY_COVERAGE.md exists from a previous iteration:
   a. Read it to understand what depth tiers were already added
   b. Include those in the coverage map (they appear as gap-fill markers in YAML files)
```

### Phase 2: Re-read UI Components

```
UI Source Resolution (per route):
  STEP A: Check if TSX exists → frontend/app/pages/{page}.tsx
    IF found: use TSX (React components — most accurate, post Phase 6)
    IF NOT found:
      STEP B: Check HTML prototypes → .claude-project/design/variations/{selected_variation}-*-{page}.html
        IF found: use HTML prototype (design output — available from Phase 3)
        IF NOT found: skip depth analysis for this route, log warning
              "No UI source found for {route} — skipping depth tiers"

  Note: At Phase 3.5 (parallel with Phase 6), TSX does not exist yet.
  HTML prototypes are the correct and expected source. Do NOT skip routes.
  HTML element extraction follows the same logic as TSX:
    - Input/textarea with maxlength or min/max attributes → Tier 9
    - Role-check conditionals, data-auth attributes → Tier 10
    - Error message elements (class="error", aria-live, etc.) → Tier 11
    - Empty state elements (class~="empty", hidden/conditional sections) → Tier 12
    - Forms with 2+ input fields → Tier 13
    - Pattern signals in element names/IDs/classes → Tier 14

For each route in routes.yaml:
  1. Resolve UI source file (TSX preferred, HTML prototype fallback — see above)
  2. Read the source and extract depth-relevant elements:

     Boundary Elements (Tier 9):
       - Input fields with maxLength={n} props
       - Input fields with type="number" and min/max props
       - Textarea fields with maxLength
       - Pagination controls (page count, total items)

     Auth Boundary Elements (Tier 10):
       - Route's auth classification from routes.yaml
       - Role-check conditionals in component (e.g., user.role === 'admin')

     Error State Elements (Tier 11):
       - useQuery hooks: extract queryKey and isError/error handling
       - useMutation hooks: extract onError callbacks
       - Conditional error renders: {isError && <div>...</div>}
       - Error boundary fallback components
       - Specific error messages in error branches

     State-Dependent Elements (Tier 12):
       - Conditional renders based on data: {items.length === 0 && ...}
       - Empty state components/messages
       - Conditional UI based on user state: {user.vehicles.length > 0 ? ... : ...}
       - Different layouts based on cart/order state

     Keyboard Elements (Tier 13):
       - Forms with 2+ input fields (candidate for Tab navigation test)
       - Forms with onSubmit handler (candidate for Enter key submission test)
       - Modal dialogs with form fields

     Bug Pattern Signal Elements (Tier 14):
       - For each category in category_detection_map (from Phase 0.7):
         Search page .tsx for code_signals
         Record: { category, signals_found: [matched keywords], pattern_ids: [matching pattern IDs] }

  3. Output: depth_inventory per route
     {
       route: "/products",
       boundary: [{ element: "search input", maxLength: 100 }],
       auth: { classification: "public", role_checks: [] },
       errors: [{ hook: "useProducts", error_message: "Unable to load products" }],
       state: [{ condition: "products.length === 0", empty_message: "No products available" }],
       keyboard: [{ form: "search form", fields: ["search input"], submit: "search button" }],
       bug_patterns: [{ category: "form_input", signals_found: ["<form", "onSubmit"], pattern_ids: ["form_input-001", "form_input-002"] }]
     }
```

### Phase 3: Gap Analysis

```
For each route in routes.yaml:
  1. Get existing coverage from Phase 1: coverage_map[route].tiers
  2. Get UI inventory from Phase 2: depth_inventory[route]
  3. For each depth tier (9-14):
     a. Check if tier is applicable (see Tier Applicability Rules)
     b. If applicable AND not in coverage_map[route].tiers:
        → Add to gap_list: { route, file, tier, elements }

  4. Also check thin-coverage pages:
     If coverage_map[route].story_count < 5
     AND depth_inventory[route] has > 3 form elements:
        → Flag as "thin coverage" for priority generation

Output:
  gap_list = [
    { file: "products.yaml", route: "/products", tier: 9, elements: [{search input, maxLength: 100}] },
    { file: "products.yaml", route: "/products", tier: 11, elements: [{useProducts, "Unable to load products"}] },
    { file: "cart.yaml", route: "/cart", tier: 12, elements: [{cart empty vs populated}] },
    ...
  ]

  gap_score = routes_with_gaps / total_routes
```

### Phase 4: Generate Stories for Gaps

For each gap in gap_list, generate stories using the tier templates below. All stories must:
- Use real data from `_fixtures.yaml` (same rules as `generate-user-stories`)
- Use canonical workflow verbs (including the new extended verbs)
- Set `auth` field correctly based on route classification
- Never include login steps in workflows (auth handled by playwright-qa-agent)

#### Tier 9: Boundary Conditions Templates

**Long text input:**
```yaml
- name: "{page} {input} handles maximum length input"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [negative, boundary]
  workflow: |
    Navigate to {route}
    Fill {input description} with 500 characters of text
    Verify {input description} is visible
    Verify {feedback element} is visible
```

**Numeric bounds:**
```yaml
- name: "{page} quantity cannot go below minimum"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [negative, boundary]
  workflow: |
    Navigate to {route}
    Fill {quantity field} with "0"
    Verify {decrease button} is disabled
```

**Pagination beyond range:**
```yaml
- name: "{page} navigating past last page shows appropriate state"
  url: "{base_url}{route}?page=999"
  auth: {auth_level}
  tags: [negative, boundary]
  workflow: |
    Navigate to {route}?page=999
    Verify {empty state or redirect indicator} is visible
```

#### Tier 10: Auth Role Boundary Templates

**Wrong role access:**
```yaml
- name: "Regular user accessing {admin_page} is denied"
  url: "{base_url}{admin_route}"
  auth: user
  tags: [negative, auth]
  workflow: |
    Navigate to {admin_route}
    Verify access denied message is visible
```

**Note:** Only generate Tier 10 stories for routes NOT already covered by `auth-guards.yaml`. Read `auth-guards.yaml` first and skip routes that already have auth guard stories.

#### Tier 11: Network/API Error Templates

**Invalid resource ID:**
```yaml
- name: "{page} with invalid ID shows error state"
  url: "{base_url}{route}/nonexistent-uuid-xyz"
  auth: {auth_level}
  tags: [negative, network]
  workflow: |
    Navigate to {route}/nonexistent-uuid-xyz
    Verify text "{korean_error_message}" appears
```

**Error state on data page:**
```yaml
- name: "{page} shows error state when API fails"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [negative, network]
  workflow: |
    Navigate to {route}
    Verify {error_element_or_message} is visible
```

Only generate Tier 11 stories when the component explicitly renders an error UI (e.g., `{isError && <p>Error message</p>}`). Extract the actual error message from the source code.

#### Tier 12: State-Dependent Templates

**Empty vs populated state:**
```yaml
- name: "{page} shows empty state message when no data"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [state, dependent]
  workflow: |
    Navigate to {route}
    Verify text "{empty_state_korean_message}" appears

- name: "{page} shows content when data exists"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [state, dependent]
  workflow: |
    Navigate to {route}
    Verify at least one {content_element} is visible
    Verify {action_button} is visible
```

#### Tier 13: Keyboard Interaction Templates

**Tab navigation:**
```yaml
- name: "{page} form fields are tab-navigable"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [accessibility, keyboard]
  workflow: |
    Navigate to {route}
    Click {first input field}
    Press Tab to focus {second input field}
    Verify {second input field} is visible
```

**Enter key submission:**
```yaml
- name: "{page} form submits via Enter key"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [accessibility, keyboard]
  workflow: |
    Navigate to {route}
    Fill {field1} with "{valid_value}"
    Fill {field2} with "{valid_value}"
    Press Enter in {last field}
    Verify {success indicator} is visible
```

#### Tier 14: Known Bug Pattern Templates

Generated from matched bug patterns in Phase 0.7. For each matched pattern, use the `prevention_spec` to derive a testable workflow.

**Form validation pattern (form_input category):**
```yaml
- name: "{page} validates {field} with proper error message"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [proactive, bug-pattern]
  workflow: |
    Navigate to {route}
    Fill {field} with {invalid_value}
    Click {submit button}
    Verify {validation error message} is visible
```

**Auth session pattern (auth category):**
```yaml
- name: "{page} handles session expiry gracefully"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [proactive, bug-pattern]
  workflow: |
    Navigate to {route}
    Verify {page content} is visible
```

**Pagination pattern (list_pagination category):**
```yaml
- name: "{page} list has pagination controls"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [proactive, bug-pattern]
  workflow: |
    Navigate to {route}
    Verify pagination controls are visible
    Verify list is sorted by most recent first
```

**Search pattern (search_filter category):**
```yaml
- name: "{page} search returns relevant results"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [proactive, bug-pattern]
  workflow: |
    Navigate to {route}
    Fill search input with "{search_term}"
    Verify search results contain "{expected_text}"
```

**File upload pattern (file_upload category):**
```yaml
- name: "{page} shows file requirements before upload"
  url: "{base_url}{route}"
  auth: {auth_level}
  tags: [proactive, bug-pattern]
  workflow: |
    Navigate to {route}
    Verify file upload area is visible
    Verify accepted file types info is visible
```

**General pattern (any category):**
Use the pattern's `prevention_spec` to derive the workflow:
1. Extract the testable user action from prevention_spec
2. Extract the expected result from prevention_spec
3. Map to canonical workflow verbs

**Dedup with _gap_patterns.yaml:** If `_gap_patterns.yaml` already covers a route+category combination, skip Tier 14 for that combination.

### Phase 5: Accumulate (Append to Files)

```
For each file that has new stories:
  1. Read the existing file content
  2. Determine iteration number:
     - If no gap-fill markers exist: iteration = 1
     - If markers exist: iteration = max(existing iterations) + 1
  3. Append after the last line of the file:

     # === ITERATION {N} GAP FILL: {comma-separated tier names} ===
     # generated: {YYYY-MM-DD} by deepen-user-stories skill

     - name: "..."
       url: "..."
       auth: ...
       tags: [...]
       workflow: |
         ...

  4. NEVER modify or delete any existing content above the marker
  5. NEVER re-add stories for tiers already covered in previous iterations
```

## Convergence Loop

When invoked with `--loop` (either standalone or from Phase 9):

```
iteration = 1
max_iterations = 8 (default, configurable via --max-iterations)
last_gap_score = 1.0
consecutive_low_gap = 0

WHILE iteration <= max_iterations:

  1. EXECUTE Phases 1-6 (full algorithm above)

  2. READ STORY_COVERAGE.md for gap_score and new_stories_added

  3. CHECK EXIT CONDITIONS (in order):

     a. NO NEW STORIES: new_stories_added == 0
        → EXIT "converged — no gaps found"

     b. SATURATED: gap_score < 0.05 AND consecutive_low_gap >= 2
        → EXIT "converged — coverage saturated"
        → Report: "Stories saturated after {N} iterations. Total: {count}. Depth score: {score}"

     c. HARD CAP: story_depth_score >= 0.95
        → EXIT "converged — depth target reached"

     d. RUNAWAY PREVENTION: total_stories > route_count * 30
        → EXIT "hard cap — max stories per route exceeded"

     e. MAX ITERATIONS: iteration >= max_iterations
        → EXIT "max_iterations reached"
        → Report: "Max iterations. Current depth score: {score}"

  4. UPDATE COUNTERS
     if gap_score < 0.05: consecutive_low_gap += 1
     else: consecutive_low_gap = 0
     last_gap_score = gap_score
     iteration += 1

END WHILE

Report final: "Deepen complete: {total_stories} stories, depth score {score}, {iterations} iterations"
```

## Error Handling

| Error | Action |
|-------|--------|
| No existing story files | FAIL — run `generate-user-stories` first |
| Page component not found for route | Skip depth analysis for that route, log warning |
| No gaps found on first run | EXIT with "already saturated" — no stories to add |
| YAML parse error in story file | Skip that file, log warning, continue with other files |
