## Issue Entry Format

Each issue in the registry follows this structure:

```markdown
| {id} | {severity} | {category} | `{file}:{line}` | {description} | {root_cause} |
```

**Field definitions:**

| Field | Description | Example |
|-------|-------------|---------|
| `id` | Sequential number | 1, 2, 3... |
| `severity` | Critical / High / Medium / Low | Critical |
| `category` | Category name (abbreviated) | API Contract |
| `file:line` | File path and line number | `services/userService.ts:42` |
| `description` | What's wrong | "Missing error handling on POST call" |
| `root_cause` | Why it's wrong | "try/catch block not wrapping httpService.post()" |

---

## Fixing Plan Entry Format

Each fix in the plan follows this structure:

```markdown
| {id} | {title} | `{file}:{line}` | {root_cause} | {suggested_fix} | {effort} |
```

**Effort estimates:**

| Level | Description | Typical Time |
|-------|-------------|-------------|
| Low | Single file change, < 10 lines | Minutes |
| Medium | Multiple files or complex logic | Hours |
| High | Architectural change or many files | Days |
