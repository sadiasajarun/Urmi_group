# Phase 3.5: User Stories (per-page authorship)

Phase 3.5 authors user-story YAMLs from the HTML pages produced by Phase 3
(design). The contract is **1:1**: every file in
`.claude-project/design/html/*.html` gets exactly one YAML in
`.claude-project/user_stories/<same-basename>.yaml`, and each YAML carries
multiple scenarios (not just a happy-path AC list).

Phase 2 (prd) no longer produces stories — HTML doesn't exist yet at that
point, so there's nothing concrete to bind stories to.

## Prerequisites

- Phase 3 (design) complete — `DESIGN_STATUS.md` must have `approved: true`
  AND `phase_complete: true`.
- `.claude-project/design/html/*.html` must be non-empty.

## Execution

Phase 3.5 is driven by `blueprints/user-stories-2.yaml`. Two agentic nodes:

1. **`split-to-per-page`** — reads `design/html/*.html`, `PROJECT_KNOWLEDGE.md`,
   `PROJECT_API.md`, `_fixtures.yaml`. For every HTML file with no matching
   YAML, creates one. For every matching YAML with fewer than 5 scenarios or
   missing scenario buckets, deepens it. Legacy feature-bundle YAMLs are split
   into their constituent pages and then retired.

2. **`evidence-check`** (deterministic) — fails the phase if any HTML page
   lacks a YAML or any YAML has fewer than 5 scenarios.

## Required scenario buckets per YAML

| Bucket | Minimum | What goes in |
|--------|---------|--------------|
| `acceptance_criteria` | 2 entries | Happy-path actions: click, submit, navigate |
| `boundary_cases` | 1 entry | Min/max, empty, whitespace, overflow |
| `error_scenarios` | 1 entry | 401/403/404/500, network failure, validation error |
| `state_transitions` | 1 entry | Loading / empty / error UI state |

**Minimum total scenarios per YAML: 5. Minimum populated buckets: 3 of 4.**

## Quality Gate

```yaml
gate: bash gates/user-stories-gate.sh {TARGET_DIR}

checks:
  - prereq-design-approved     # DESIGN_STATUS.md approved:true
  - html-pages-exist           # design/html/*.html > 0
  - story-files-exist          # user_stories/*.yaml > 0
  - page-parity                # yaml_count >= html_count * 0.95
  - naming-parity              # every HTML basename has a matching YAML basename
  - scenarios-per-story        # every YAML has >= 5 scenarios
  - bucket-coverage            # every YAML has >= 3 of 4 scenario buckets
  - no-duplicate-ids           # no duplicate story ids across files
```

## Loop Integration

- **Command**: `fullstack-2 --phase user-stories --loop`
- **Max iterations**: 5 (default; override with `--max-iterations`)
- **Convergence criteria** (checked after each iteration):

```
1. PARITY HIT: yaml_count >= html_count AND all YAMLs >= 5 scenarios
   → EXIT "converged — per-page parity reached"

2. NO PROGRESS: yaml_count unchanged for 2 consecutive iterations
   → EXIT "stagnation — agent not converging"

3. MAX ITERATIONS: iteration >= max_iterations
   → EXIT "max iterations reached"
```

## Auto-Trigger from Phase 9

`test-browser-2` blocks on `user-stories-check` (pre-flight deterministic
node). If stories are missing or parity drops below 95%, Phase 9 aborts and
the user must re-run Phase 3.5 first.

## Phase Completion — Status Update

After the gate passes, update
`{TARGET_DIR}/.claude-project/status/{project}/PIPELINE_STATUS.md`:

1. **Progress Table** — Set `user-stories` row: Status=`Complete`,
   Score=gate score, Output=`N YAMLs for N HTML pages`.
2. **Execution Log** — Append `| {DATE} | user-stories | {GEN} | {DURATION} | {RESULT} | {SCORE} | parity: yaml/html |`.
3. **Phase Details** — Record final yaml_count + html_count under
   `### user-stories`.
4. **Artifact Hashes** — `sha256sum user_stories/_index.yaml`.
5. **Configuration** — `last_run = <ISO>`, refresh `pipeline_score`.
