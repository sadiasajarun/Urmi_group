# /fullstack-pm — PM Pipeline (P1–P3)

PM-track orchestrator. Runs only the PM slice of the fullstack pipeline: requirements interview (P1), PRD generation (P2), and design prototypes (P3) up through client approval and full HTML generation. Stops after P3g.

## Handoff to /fullstack-dev

After this command completes, `/fullstack-dev <project>` becomes runnable. PM produces exactly two handoff artifacts (plus supporting files):

1. `.claude-project/docs/PRD.md` — canonical PRD (also archived in `.claude-project/prd/{PROJECT}_PRD.md`)
2. `.claude-project/design/html/<role>/*.html` — approved HTML, organized by role folder
3. `DESIGN_STATUS.md` with `approved: true`, `phase_complete: true`, and PRD snapshot fields (used by /fullstack-dev Tier 2 consistency check)

Everything else the Dev side needs (PROJECT_KNOWLEDGE/API/DATABASE.md, CLAUDE.md, backend/frontend scaffolds, user stories) is generated by `/fullstack-dev` — not by PM.

## PM Phase List

| Logical | Physical file | What it does |
|---|---|---|
| **P1-spec** | `pipeline/phases/00-spec.md` (reused) | Requirements interview → `.claude-project/status/{project}/seed-*.yaml` |
| **P2-prd** | `pipeline/phases/pm-2-prd.md` (new) | Generate PRD only → `docs/PRD.md` (snapshot to `prd/history/v{N}`) |
| **P3-design** | `pipeline/phases/pm-3-design.md` (new) | 3 variations → client picks → role-folder HTML + snapshot |

## HARD STOP — PM-track scope enforcement

This command runs ONLY P1-spec, P2-prd, P3-design.

**Forbidden:**
- Reading or invoking any of: `01-init.md`, `02-prd.md`, `03-design.md`, `03.5-user-stories.md`, `04-database.md`, `05-backend.md`, `06-frontend.md`, `07-integrate.md`, `08-test-api.md`, `09-test-browser.md`, `10-ship.md`
- Reading or invoking any `dev-*.md` phase module
- Writing to `CLAUDE.md`, `.claude-rules`, `backend/`, `frontend/`, `memory/`, `.claude-project/customization/`, `.claude-project/docs/PROJECT_*.md`, `.claude-project/user_stories/`
- Updating PIPELINE_STATUS.md rows for any Dev phase (D1–D10)

If a user explicitly passes `--phase <name>` with a Dev phase name (e.g., `D5-backend`), reject with:

```
Error: Phase "<name>" is not in the PM track (P1-spec, P2-prd, P3-design).
Use /fullstack-dev for Dev phases, or /fullstack for the legacy full pipeline.
```

## Execution Instructions

### Step 1: Parse Arguments

```
project = $1
action = --run | --run-all | --loop | --phase <name> [--loop] | --resume | --reset <name> | (none = status)
skip_spec = --skip-spec
prd_path = --prd <path>
target_path = --path <dir>
```

If no project name provided → ask via AskUserQuestion.

### Step 2: Resolve TARGET_DIR

```
IF --path: TARGET_DIR = absolute(path), mkdir -p if missing
ELSE:      TARGET_DIR = cwd
```

### Step 3: Resolve active phase group

- Load `.claude/pipeline/phase-groups.yaml`
- Active group = `pm` = [P1-spec, P2-prd, P3-design]
- Keep this in memory; reject any `--phase <name>` not in this list

### Step 4: PRD pre-read (if --prd provided)

Same as `commands/fullstack.md` Step 1.6 — split PRD into `PRD_chunk_*.md` (400-line chunks) under `.claude-project/context/` for A-02 prevention. Reuse legacy logic directly.

### Step 5: Status file

If `.claude-project/status/{project}/PIPELINE_STATUS.md` doesn't exist, create minimal one with P1-spec / P2-prd / P3-design rows (Pending). If this file exists from a legacy `/fullstack` run, **do not overwrite** — append P-row placeholders if absent, warn the user about legacy/neo mixing (see below).

### Step 6: Legacy-mixing detection

Warn if legacy artifacts are present on this project:
- `.claude-project/docs/PROJECT_KNOWLEDGE.md` (without our frontmatter)
- `.claude-project/design/html/*.html` as flat (no role subfolders)
- PIPELINE_STATUS.md has rows named `prd`, `design`, etc. (legacy keys)

Print:
```
WARNING: Legacy /fullstack artifacts detected on this project.
         Running /fullstack-pm may leave the project in a mixed state.
         Recommended: use /fullstack for legacy projects, or fully migrate before continuing.
```
Proceed only if user passes `--force`.

### Step 7: Action handler

- **No action** (bare `/fullstack-pm <project>`) → show status (P1-spec / P2-prd / P3-design row snapshots)
- **`--run`** → run next pending PM phase once
- **`--run-all`** → run P1 → P2 → P3 sequentially; stop at P3d pause if `approved:true` missing
- **`--loop`** → infinite-loop PM phases until gates converge or max-generations hit
- **`--phase <P1-spec|P2-prd|P3-design>`** → run that specific phase. If `--loop`, loop it.
- **`--resume`** → after manual DESIGN_STATUS.md edit (approved:true recorded), continue P3 from current sub-step (P3e → P3f → P3g)
- **`--reset <phase>`** → set PM phase row to Pending (must be a PM phase)

### Step 8: Phase execution

For each phase to execute:

1. Load module: `pipeline/phases/<file>.md` (as mapped in `phase-groups.yaml`)
2. Load manifest: `pipeline/manifests/<phase-name>.manifest.yaml`
3. Pre-hydrate: Read files declared in manifest's `pre_hydrate`
4. Execute blueprint: `blueprints/<phase-name>.yaml` (walk nodes: deterministic → agentic → gate)
5. Run gate: `gates/<phase-name>-gate.sh {TARGET_DIR}` (for P3 only — P1/P2 use evaluation-only)
6. Update PIPELINE_STATUS.md row with logical name (**P1-spec / P2-prd / P3-design**, NOT spec/prd/design)
7. If gate fails → report, return non-zero

### Step 9: P3d Pause Handling

If P3-design execution aborts at `require-variation-selection` node (DESIGN_STATUS.md `approved:true` missing), present user with:

```
⏸ Design review required.

We generated 3 design variations (A, B, C) with {N} representative pages each.
  .claude-project/design/variations/showcase-ALL.html

Steps:
  1. Open showcase and select a variation (A, B, or C, or a Mix)
  2. Edit .claude-project/status/{project}/DESIGN_STATUS.md (or .claude-project/design/DESIGN_STATUS.md):
     selected_variation: <A|B|C>
     approved: true
     approved_at: <ISO8601>
  3. Re-run: /fullstack-pm {project} --phase P3-design --resume
```

### Step 10: Loop mode

Same infinite-loop semantics as `commands/fullstack.md`, scoped to PM phases only. On convergence or max-generations, stop and print summary.

## Example Usage

```bash
# Full PM pipeline from scratch
/fullstack-pm my-saas --run-all

# Provide existing PRD
/fullstack-pm my-saas --run-all --skip-spec --prd ./docs/MyProduct_PRD.md

# Run PRD only
/fullstack-pm my-saas --phase P2-prd

# After client picks variation B and edits DESIGN_STATUS.md
/fullstack-pm my-saas --phase P3-design --resume

# Iterate design until QA converges
/fullstack-pm my-saas --phase P3-design --loop

# Update with new PRD
/fullstack-pm my-saas --update --prd ./docs/MyProduct_PRD_v2.md

# Check status
/fullstack-pm my-saas
```
